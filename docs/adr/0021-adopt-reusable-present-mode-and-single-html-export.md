# ADR-0018: Adopt reusable Present mode and single HTML export

- Status: accepted
- Date: 2026-05-07

## Context

Starry Slides has an editor-facing `Present` button and an export menu entry for
`Single HTML`, but neither is implemented as a real presentation runtime. The
product now needs one Present mode that works from two entry points:

- inside the editor, by clicking the header `Present` button
- outside the editor, by opening an exported single HTML file

The implementation must not fork the presentation behavior. Presenter controls,
slide navigation, laser pointer, temporary ink, and exit behavior should be
implemented once and then adapted to each host. The editor host can use React
state and already-loaded `SlideModel` objects. The exported HTML host cannot
depend on Vite, React bundles, or network requests, so it needs a self-contained
runtime generated at export time.

Related constraints:

- ADR-0008 keeps the product inside one package with internal core/editor/runtime
  boundaries.
- ADR-0015 keeps editor chrome compact and neutral, while generated slide
  content remains visually independent.
- ADR-0016 established the export split: core owns selection/planning,
  runtime owns file generation, CLI/editor expose entry points.
- ADR-0017 introduces deck-level `hidden` metadata. Presentation export should
  skip hidden slides by default when presenting the whole deck.

## Decision

Adopt a reusable Present mode and single HTML export pipeline.

Present mode behavior:

1. Clicking the editor header `Present` button enters Present mode directly.
2. Present mode renders only the active slide set and presentation chrome. The
   slide occupies the largest possible viewport while preserving slide aspect
   ratio.
3. A bottom floating toolbar overlays the slide, auto-hides while idle, and
   reappears on pointer movement, focus, or interaction near the bottom.
4. The toolbar supports previous/next slide navigation, `current / total`
   pagination, laser pointer mode, pen mode, pen color selection, and exit.
5. Laser pointer mode makes the pointer visibly high contrast over slide
   content and shows a fluorescent cursor marker.
6. Pen mode draws temporary ink over the current slide. Ink is presentation
   state only; it must not mutate `SlideModel.htmlSource` or saved slide files.
7. Switching away from pen mode or pressing `Escape` exits pen mode. Pressing
   `Escape` outside pen mode exits Present mode in the editor host.
8. Hidden slides are skipped in the presentation order. If every slide is hidden,
   the host falls back to the full deck so the presenter is not empty.

Implementation structure:

1. `src/editor/components/presenter-view.tsx` owns the reusable React presenter
   component used by the editor.
2. `src/core/html-export.ts` owns browser-safe selection, sanitization, and
   single-file document generation helpers.
3. `src/runtime/html-export.ts` loads deck files and writes the exported HTML
   file.
4. `src/cli/index.ts` exposes `starry-slides export html [deck] --out <file>`.
5. `vite.config.ts` exposes an editor middleware route for single HTML export,
   mirroring the existing PDF route shape.
6. `src/editor/app/use-slides-data.ts` exposes `exportHtml`, flushes pending
   saves, downloads the returned HTML blob, and keeps header export behavior as
   a trigger only.

The exported single HTML file must open directly into Present mode without a
separate editor shell, manifest fetch, or dev server. It may use embedded CSS
and plain JavaScript generated by `src/core/html-export.ts`.

## Implementation Plan

- **Affected paths**:
  - `src/core/html-export.ts`
  - `src/core/index.ts`
  - `src/runtime/html-export.ts`
  - `src/cli/index.ts`
  - `vite.config.ts`
  - `src/editor/index.tsx`
  - `src/editor/app/use-slides-data.ts`
  - `src/editor/components/editor-header.tsx`
  - `src/editor/components/presenter-view.tsx`
  - `src/editor/styles/index.css`
  - `src/core/html-export.test.ts`
  - `src/cli/index.test.ts`
  - `e2e/tests/presenter-mode.spec.ts`
  - `docs/adr/README.md`

- **Pattern**:
  - keep deck selection and exported document generation in `src/core`
  - keep filesystem access in `src/runtime` and Vite middleware
  - keep editor header callbacks thin; they should call `onPresent` or export
    callbacks, not implement presentation/export logic
  - share interaction semantics between React presenter and exported HTML
    runtime, even though exported HTML uses plain JavaScript
  - do not persist laser pointer state, pen mode, pen color, or ink strokes

- **Tests**:
  - core tests cover hidden-slide filtering, fallback when all slides are hidden,
    escaping of embedded slide HTML, and generated HTML markers
  - CLI tests cover `export html` command parsing, static verification, output
    file creation, and JSON result shape
  - E2E tests cover editor Present entry, slide navigation and pagination,
    toolbar auto-hide/reveal, laser pointer mode, pen drawing and Escape exit,
    color picker reuse for pen color, editor exit, and exported HTML opening in
    Present mode

## Verification

- [ ] editor `Present` enters a full-window Present mode without sidebar or
      editing toolbar chrome
- [ ] slide frame is the largest visible surface and keeps its aspect ratio
- [ ] bottom toolbar auto-hides and reappears on interaction
- [ ] previous/next buttons update slide content and pagination
- [ ] laser pointer mode shows a visible fluorescent cursor
- [ ] pen mode draws temporary ink, supports color selection, and exits on
      `Escape` or mode switch
- [ ] exiting Present mode returns to the editor at the selected slide
- [ ] `starry-slides export html [deck] --out <file>` writes one self-contained
      HTML file that opens directly in Present mode
- [ ] editor single HTML export downloads the same runtime format through the
      local middleware route
- [ ] hidden slides are skipped unless all slides are hidden
- [ ] `pnpm lint`, `pnpm test`, `pnpm build`, and relevant E2E tests pass

## Consequences

Presentation behavior becomes a first-class reusable runtime instead of a button
placeholder. Editor presentation and exported presentation still have different
hosts, but they share one documented behavior contract and core export helpers.

The exported HTML file intentionally duplicates a small plain-JavaScript
presenter runtime instead of depending on the editor React bundle. This keeps the
artifact portable and offline-friendly at the cost of maintaining parity tests
between the React presenter and generated runtime.

HTML export becomes the second real export format after PDF, so future export
work should reuse the ADR-0016 split: core plans and validates, runtime writes,
CLI/editor expose.

## Alternatives considered

### Open browser fullscreen with the existing editor iframe

Rejected. That would keep presentation behavior coupled to editor chrome and
would not satisfy the standalone single HTML export requirement.

### Export the whole React app into one HTML file

Rejected. The exported artifact should be a presentation runtime, not an offline
copy of the editor. Bundling the editor would increase artifact size and make
offline behavior dependent on build tooling details.

### Build only a standalone HTML presenter first

Rejected. The editor `Present` button is a required entry point and should not
drift from exported presentation behavior.

### Persist pen drawings into slide HTML

Rejected. The requested pen is temporary presentation ink. Persisting it would
turn Present mode into an editing surface and conflict with the existing
operation/history model.
